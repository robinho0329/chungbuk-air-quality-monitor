# 🔍 코드 리뷰 기준 (Code Review Criteria)

> 사람 리뷰어와 AI 리뷰어(`.claude/agents/qa-reviewer.md`, `scripts/daily_dev_loop.py`)가
> **동일한 기준**으로 코드를 검토하기 위한 단일 기준 문서.
> 코드 리뷰는 "틀린 사람을 찾는 절차"가 아니라, 더 안전하고 오래 유지되는 코드를 만드는 협업 절차다.

---

## 1. 리뷰 4대 축

| 축 | 확인 포인트 |
|----|------------|
| 🔒 **보안** | API Key/시크릿이 코드에 하드코딩되지 않았는가 · `.env`가 `.gitignore`에 포함되는가 · 개인정보/민감값이 프롬프트·로그에 노출되지 않는가 · 파일 삭제·시스템 명령 등 위험 코드가 없는가 · 외부 입력값을 그대로 신뢰하지 않는가 |
| ⚡ **성능** | 같은 파일·API를 불필요하게 반복 호출하는가 · 같은 계산을 중복하는가 · 데이터가 커져도 동작 가능한 구조인가 · 응답이 느려질 가능성이 있는가 |
| 📖 **가독성** | 변수·함수명이 의미를 전달하는가 · 함수 하나가 한 가지 역할만 하는가 · 주석이 부족하거나 과하지 않은가 · 코드 흐름이 읽히는가 |
| 🔧 **유지보수성** | 중복 코드가 함수로 묶였는가 · 긴 함수가 역할 단위로 분리됐는가 · 예외 처리(파일 없음·잘못된 입력·JSON 오류)가 있는가 · 작은 단위로 테스트 가능한가 |

> 우선순위 원칙: **정확성 → 가독성 → 안정성**을 먼저 확보한 뒤 성능을 최적화한다.
> 과도한 최적화·복잡한 패턴보다 "다음 사람이 이해하기 쉬운 개선"을 우선한다.

## 2. AI 리뷰 출력 형식

AI 리뷰는 항상 아래 표로 정리하고, 마지막에 **"우선 반영할 3가지"**를 별도로 제시한다.

```
| 위치(함수/파일) | 문제 유형 | 심각도 | 개선 제안 | 이유 |
|----------------|----------|--------|----------|------|
```

- **문제 유형**: 버그 / 보안 / 성능 / 가독성 / 구조 / 예외처리 / UX
- **심각도 기준**

| 심각도 | 기준 | 예시 |
|--------|------|------|
| 🔴 높음 | 실행 오류 · 데이터 손실 · 보안 | 수집 실패, API Key 노출, DB 손상 |
| 🟡 중간 | 유지보수성 저하 | 함수가 너무 김, 중복 코드 |
| ⚪ 낮음 | 스타일·취향 | 출력 문구, 줄 간격 |

→ **높음**부터 반영하고, **낮음**은 보류 가능.

---

## 3. AI 리뷰의 한계 (반드시 검증)

- **환각**: 존재하지 않는 함수·라이브러리를 말하거나 실제 코드와 안 맞는 지적을 할 수 있다.
- **실행환경 미인식**: 파일 경로·OS·패키지 버전·권한 등 실제 환경에서만 드러나는 문제를 놓친다.
- **과도한 개선**: 프로젝트 목표보다 복잡한 구조를 제안할 수 있다.

> 원칙: AI 출력은 **정답이 아니라 검토 후보 목록**이다.
> 항상 실행·테스트·리뷰로 검증하며, **최종 책임은 개발자에게 있다.**

---

## 4. 리뷰 → 리팩토링 흐름

1. 리뷰 결과를 **개선 목록**(번호 / 항목 / 우선순위 / 반영여부)으로 변환
2. 우선순위 높은 것부터 **작은 단위로** 반영 ("전체 마음대로 바꿔줘"가 아니라 "X만 분리, 기존 기능 변경 금지")
3. 수정 후 **항상 실행 + 회귀 테스트** (`pytest`)
4. 변경 내용·이유·테스트를 **PR 설명**([PULL_REQUEST_TEMPLATE](../.github/PULL_REQUEST_TEMPLATE.md))에 정리
